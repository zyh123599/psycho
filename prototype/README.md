# 心潮 V0.6：纯前端多模态画像与画像混合叙事

当前版本没有应用后端。用户在「我的」中配置自己的 OpenAI 兼容 API，浏览器直接调用模型；任何持久化数据都只写在当前设备的 `localStorage`。

## 运行

在仓库根目录执行：

```bash
npm ci
npm run dev
```

```bash
npm run test:frontend
npm run build
```

不要把 API Key 写入源码、`.env` 或构建参数。请在运行后的「我的 → 自定义模型 API」中保存。供应商兼容要求与结构化输出契约见 [`CUSTOM_API.md`](CUSTOM_API.md)。

## 产品流程

底部导航为「今天 / 对话 / 卡槽 / 回响 / 我的」。完整章节路径为：

1. 用户写至少两张闪念，可选最多四张 JPEG/PNG/WebP 图片；
2. 用户确认图片处理权利并单独授权持续画像；
3. 本地规则先立即给出主题候选，模型画像在后台异步更新；
4. 用户确认主题后，模型结合主题与最新本机画像生成六张叙事卡；
5. 卡片文案由模型生成，潮向、信号和满潮规则保持本地固定，避免模型控制评分逻辑；
6. 对话、行动、未来回响与日报可读取最新文字画像；
7. 未配置、未授权、CORS 失败、超时或模型输出无效时，自动保留内置叙事与本地对话降级。

## 持续多模态画像

图片不是单独保存的“画像附件”。模型在同一次请求中读取用户授权的文字和图片，返回结构化的 `multimodal_observations`，其中只包含：

- 参与观察的图片/文字 source ID；
- 图片或跨模态的文字观察；
- 该观察对暂时性画像的贡献；
- 不确定性。

原始图片始终只存在当前页面内存。图片处理完成后，只有上述文字化观察随画像写入本机，后续文字更新通过独立的 `previous_profile_context` 延续这些观察。

旧画像不会再伪装成新的第一人称文本证据。新文本、互动和图片位于 `fresh_evidence`，旧画像只作为可被覆盖的连续性上下文；提示词和本地净化还会移除“本次没有新增第一人称近况”等流水线元话术。

## 异步与竞态

- 画像端点不阻塞闪念、主题、选卡、聊天、行动或完成页；
- 同一时刻最多一个画像请求，请求期间的新变化合并为下一次快照；
- 本地证据指纹只保存哈希，不保存输入原文；相同证据不会重复扣费；
- 每张图片有稳定 source ID 和签名，已处理图片不会在同一流程重复上传；
- 删除画像或撤销授权会 abort 在途请求，并通过 generation token 阻止迟到结果复活；
- 叙事生成也可被新画像替换，但用户开始答题后不会再异步换卡。

## 本地存储键

| 键 | 内容 |
| --- | --- |
| `xinchao.custom-api.v1` | 自定义 Base URL、API Key、模型名、图片精度 |
| `xinchao.ai-consent.v1` | AI 与持续画像授权开关 |
| `xinchao.reflective-profile.v1` | `local_profile_version: 2.0` 的最新多模态文字画像及证据指纹 |
| `xinchao.tide-cards.v1` | 内置潮笺 ID 与收藏时间 |
| `xinchao.future-echoes.v1` | 用户主动保存的未来回响 |
| `xinchao.report-feedback.v1` | 日报“有帮助 / 不太像我”的低粒度反馈 |

画像键名保留 `v1` 是为了就地迁移已有本机数据；实际内容版本已升为 `2.0`。无法净化的旧画像会被忽略。

## 隐私与安全事实

- 心潮自身不设置后端或云端画像仓；
- 模型服务商仍会收到每次明确授权发送的文本、图片或对话，服务商如何记录由其政策和配置决定；
- API Key 保存在前端并非秘密保险箱，同源脚本、扩展或设备使用者可能读取；
- 聊天内置危机词预检，模型也返回结构化安全提示，但本产品不能代替紧急服务；
- 不从人脸、表情、身体、衣着或外貌推断情绪、人格、诊断或敏感属性；
- 画像、建议和叙事均应被视为暂时、可纠正的反思材料。

## 浏览器兼容

自定义服务商必须正确响应 CORS `OPTIONS` 预检，并允许 `Authorization`、`Content-Type`、`GET`、`POST`。HTTPS 前端不能调用 HTTP API。若服务商不满足这些条件，纯浏览器前端无法绕过；应由服务商添加 CORS/HTTPS，或在 Capacitor 原生包中另行采用经过审计的原生网络实现。
